OPENi Permissions

Before an application can write to a users cloudlet it must first set permission within the users cloudlet.

The examples used within this tutorial should all be already configured on Demo2. They are also at the bottom of this tutorial.

This is done using the POST Permissions endpoints within the Cloudlet API.

[
  {
    "type": "type",
    "ref": "t_195224462830d4175a9b92301f81fc59-159",
    "access_type": "CREATE",
    "access_level": "APP"
  },
 {
    "type": "type",
    "ref": "t_195224462830d4175a9b92301f81fc59-159",
    "access_type": "READ",
    "access_level": "APP"
  },
 {
    "type": "type",
    "ref": "t_195224462830d4175a9b92301f81fc59-159",
    "access_type": "UPDATE",
    "access_level": "APP"
  },
 {
    "type": "type",
    "ref": "t_195224462830d4175a9b92301f81fc59-159",
    "access_type": "DELETE",
    "access_level": "APP"
  }
]

All the permissions must be posted in an array. The permission above allow the clientID for this Auth Token to perform all CRUD actions for the specified type upon the users cloudlet.

The permissions can also have an access level of cloudlet and a type of object as shown below.

{
      "ref": "00000001-5203-4f5b-df3e-7f06c795775d",
      "type": "object",
      "access_level": "CLOUDLET",
      "access_type": "UPDATE"
    }
